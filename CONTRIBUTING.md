# Contributing

This document covers the build, the test loop, and how PRs land.

## Build and test locally

You need Docker 20.10+, Go 1.24+, GNU Make, and `git`.

```
git clone https://github.com/rachitkumar205/atlantis
cd atlantis
cp .env.example .env
make codegen        # generates gen/, clients/go/, atlantis/ from testdata/schema/
make dev-isolated
```

The generated tree is gitignored; `make codegen` produces it. Rerun after editing `.atl` fixtures or codegen emitters.

`make dev-isolated` starts Postgres, memcached, and the server in containers on a throwaway volume (versions pinned in `docker-compose.dev.yaml`). Verify the server is up:

```
curl localhost:8080/healthz
```

### Run the tests

```
make test                  # unit tests only (fast)
make test-integration      # spins up testcontainers; requires Docker
make test-all              # both
```

A single package or test:

```
go test ./internal/dsl/...
go test ./internal/dsl/ -run TestParser
```

The unit tests live alongside the code they test. Integration tests live under `tests/integration/`.

### Iterate on the codegen

```
make codegen               # regenerates from the bundled schema fixtures
make test                  # unit tests on the codegen and the emitted code
make codegen-check         # fails CI if regenerated output differs from committed
```

`make codegen-check` is the gate that keeps the committed generated tree in sync with what the codegen would emit. Run it before opening a PR.

Schema fixtures live under `testdata/schema/`. To exercise a codegen change end-to-end, edit a fixture, run `make codegen`, and inspect the diff under `gen/`.

## Where the code lives

| Directory | What |
|---|---|
| `cmd/server/` | Server entry point |
| `cmd/tide/` | Caller CLI |
| `cmd/tidectl/` | Admin CLI |
| `internal/dsl/` | Lexer, parser, AST, IR, validator |
| `internal/codegen/` | Emitters: proto, server, client, SQL |
| `internal/codegen/coltype/` | Shared column-type mapping |
| `internal/introspect/` | Live-database introspection (consumed by adopt) and the unique-index drift detector |
| `internal/storage/pg/` | Postgres connection, transactions, outbox |
| `internal/cache/` | Memcached client, read path, outbox worker |
| `internal/runtime/` | Helpers linked into generated code |
| `clients/go/` | Generated Go SDK module |
| `tests/integration/` | End-to-end tests against real Postgres + memcached |
| `migrations/infra/` | Hand-written infrastructure migrations |
| `testdata/schema/` | `.atl` fixtures the codegen tests/CI build against |

[Architecture overview](docs/architecture/) covers how the pieces fit.

## Style

Go code follows `gofmt` and `golangci-lint` defaults. The lint config is in `.golangci.yaml`.

Comments explain non-obvious invariants and design decisions. Multi-paragraph block comments are appropriate when the rationale isn't obvious from the code.

Generated files carry a `// Code generated by tidectl. DO NOT EDIT.` header. Editing one is a lint failure; instead, change the emitter and regenerate.

## PR process

For small fixes (a typo, a missed nil check, a one-line bug) — just open the PR. CI runs the full test suite plus `make codegen-check`; passing CI is the bar.

Open an issue first for non-trivial work so the design can be agreed before code lands.

PRs should:

- Stay scoped. One thing per PR.
- Include tests for new behavior. Bug fixes include a regression test.
- Avoid drive-by reformatting in unrelated files.
- Have a commit message that explains the rationale.

## What we won't merge

- Changes that re-introduce a schema-mutation path outside the `tide apply` flow. See [Schema as code](docs/concepts/schema-as-code.md).
- A `Backend` interface abstraction over PostgreSQL. See [Architecture](docs/architecture/) for the rationale.
- Changes to the generated tree without the corresponding emitter change.
- Vendored copies of third-party code unless there's a clear reason `go mod` can't handle it.
